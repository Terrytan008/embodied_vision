## 第一幕：颠覆性定义——我们到底在创造什么？

### 1.1 重新定义产品本质

**我们创造的不是"传感器"，而是一套"具身触觉的视觉等价物"。**

现有行业将双目相机定义为"获取深度信息的传感器硬件"，这是根本性的认知误区。在具身智能语境下，开发者真正需要的是：**机器人能够理解其所处环境、并做出物理交互决策的视觉能力**。

两者的根本差异：

| 传统双目相机 | 具身视觉解决方案 |
|-------------|----------------|
| 输出：距离数组 | 输出：可供决策的空间语义 |
| 标定：出厂一次性 | 标定：运行时自标定 + 场景适应 |
| 失效：输出垃圾数据 | 失效：主动上报置信度 + 备援策略 |
| 适配：调参工程师 | 适配：开箱即用的 API |

换言之，我们卖的是**机器人看世界的方式**，不是一只眼睛。

### 1.2 被忽视的感知维度——真正的破局点

业界内卷分辨率（2K→4K）和帧率（30→120fps），却忽视了以下三个真正限制具身机器人落地的维度：

**破局点一：光照鲁棒性的"暗能力"**
> 当前所有双目方案在从阳光直射的室外进入室内阴影区时，深度误差从±1cm跳变到±50cm以上——这对室内外无缝操作的机器人是致命的。

差异化方向：**主动结构光 + 被动立体匹配双模切换**。在光照骤变时降级为短基线主动模式，而非让深度值彻底失效。

**破局点二：材质感知能力**
> 传统双目系统将玻璃、镜面、透明塑料袋一律处理为"无法匹配"，直接放弃这些区域的深度估计。但真实仓库中，塑料袋包装、玻璃门、镜面货架恰恰是最密集的障碍物。

差异化方向：**Specular Highlight Detection（镜面高光检测）** + 材质自适应匹配窗口，让系统在遇到高光区域时输出"高光区域深度不可靠"的语义标注，而非盲目插值。

**破局点三：功耗-热-算力的三体约束**
> 高性能双目方案需要15-30W算力，导致机器人在散热设计上必须妥协。在人形机器人上尤为致命（头部散热困难）。

差异化方向：**异步事件相机融合**。在静态或低速场景下，事件相机功耗<100mW，仅在检测到运动目标时唤醒全功耗立体匹配模块。

### 1.3 开发者三大"尖叫"痛点

**痛点一："买回来三个月还在调标定"**
> 当前所有双目模组都依赖繁琐的工厂标定流程。一旦机械结构因振动或碰撞产生微变形，标定参数立即失效，需要返厂或高价工程师重新标定。
>
> **开发者真正需要**：开箱即用，且在使用过程中自动进行在线标定纠偏。Thor平台<1秒收敛，让这个承诺真正可信。

**痛点二："你的SDK跟我的ROS2版本不兼容"**
> 几乎所有硬件厂商的SDK都以闭源黑盒形式交付，与ROS2的版本迭代形成永恒的兼容地狱。
>
> **开发者真正需要**：开放ROS2标准接口源码（而非闭源SDK），以及跨版本兼容能力。

**痛点三："你的深度图在运动时是错的，但我不知道哪帧是错的"**
> 现有产品在机器人快速移动时都会产生运动畸变，现有产品要么不告知用户这个问题，要么让用户自己去做IMU后融合校正，门槛极高。
>
> **开发者真正需要**：系统在输出深度图的同时，输出每一帧的"置信度热力图"和"运动畸变校正量"，让决策层自己判断如何使用这些数据。

## 第二幕：不可妥协的绝对原则

### 2.1 安全即架构

**物理层要求：**

- **硬件同步双播机制**：主从Sensor之间采用硬件触发同步（HSYNC脉冲，精度<10ns），而非软同步。
- **看门狗双层防护**：Sensor ISP内部集成独立看门狗，每100ms强制检查一次I2C通信状态。若检测到单次通信超时，ISP立即冻结最后一帧有效数据并置位错误标志，**绝不输出未经校验的随机数据**。
- **深度值白名单机制**：任何深度值在输出前必须通过白名单校验（范围：0.1m–10m，变化率<5m/s）。超出白名单的数值直接替换为"无效"并上报异常。

**固件层要求：**

```c
// 深度值安全校验伪代码
bool safe_depth_output(float depth, float prev_depth, uint64_t timestamp) {
    if (depth < DEPTH_MIN || depth > DEPTH_MAX) return false;  // 范围校验
    float delta = fabsf(depth - prev_depth);
    float dt = (timestamp - prev_timestamp) / 1e9;
    if (dt > 0 && (delta / dt) > MAX_DEPTH_VELOCITY) return false;  // 物理可行性校验
    return true;  // 通过校验，输出
}
```

**底线**：任何单一模块（Sensor、ISP、SoC、线缆）的故障，系统的输出永远不会是"导致机器人撞墙的危险深度值"。

### 2.2 开放与护城河的精妙平衡

**开源层（获取信任与生态）：**

```
开源内容：
├── ROS2 驱动源码（完整，含完整 ROS2 Humble/Jazzy 支持）
├── 在线标定算法（张正友棋盘格 + 自研在线纠偏）
├── 双目匹配核心算法（Semi-Global Matching，开源）
├── 置信度图算法（开源）
├── 运动畸变校正算法（IMU-视觉紧耦合，开源）
└── 完整标定数据集（1000+组，已脱敏）
```

**护城河层（商业壁垒）：**

```
商业壁垒：
├── 核心IP：FPGA/ASIC 中的极低功耗深度计算引擎（自主设计，不对外出售RTL）
├── 差异化硬件：定制MEMS结构光投影模组（不透光外壳内嵌，无法逆向）
├── 服务壁垒：云端标定数据库（基于数万台设备的数据积累，持续优化默认参数）
└── 独家集成：与 NVIDIA Jetson/DRIVE 平台的深度优化（官方合作，含源码级集成）
```

**开源→护城河的正向循环飞轮：**

```
开发者越多 → 社区PR贡献 → 优质算法迭代 → 生产数据回流云端 →
云端标定数据库优化 → 商业模组性能领先30-50% → 更多开发者购买商业版
```

## 第三幕：成功的"感觉"与"证据"

### 3.1 产品美学哲学

> **"像影子一样存在，像眼睛一样可靠。"**

开箱即用，无需复杂调参，API调用如本能反应一样自然。开发者不应"感受到"硬件的存在——它应该像影子一样融入机器人的行为逻辑中：永远在正确的时间给出正确的信息，但从不刷存在感。

## 第四幕：战略路径推演

### 4.1 技术路径选择

**最终推荐方案：被动双目立体视觉 + 轻量化神经网络匹配加速 + 内置6轴IMU紧耦合**

| 备选路径 | 主要缺陷 | 为何不适合"爆款" |
|---------|---------|-----------------|
| 主动结构光 | 室外阳光直射失效；专利壁垒高；成本高30% | 不符合"室内外无缝"差异化 |
| 事件相机融合 | 生态极不成熟；算法门槛高；开发者难以接受 | 6个月内无法建立开发者生态 |
| 仿生光学 | 量产工艺不成熟；算法重写代价极高 | 护城河不明晰，供应链风险极高 |
| 单目+Learning | 泛化能力差；需要大量标注数据；实时性差 | 无法保证"安全即架构"底线 |

### 4.2 首发场景：仓储AMR透明物体抓取挑战

> **仓储移动机器人（AMR）在混乱仓库中识别并抓取透明塑料袋、塑料袋包裹、镜面罐头，同时在阳光直射的天窗下与阴影区域之间无缝穿行。**

**为什么这个场景具有极致病毒传播力：**

| 维度 | 对比 |
|------|------|
| 目标观众 | 服务机器人圈（家用）→ 工业机器人圈 + AMR圈（盘子更大）|
| 商业价值 | 家用（慢）→ 仓储（快，付费意愿强）|
| 内容传播 | 病毒视频（ToC）→ 工厂实际部署案例（ToB传播更快）|
| 技术壁垒展示 | 玻璃检测 → 透明物体+材质分类+运动畸变校正三重叠加 |
| 竞品对比说服力 | 够用 → **碾压性优势**（现有AMR在透明物体前必须依赖额外传感器）|

### 4.3 最大风险与早期预警机制

**风险一：在线标定算法的收敛速度**

### A.1 双目基线距离

**深度精度公式：**

```
δZ = (Z² × δp) / (f × B)

参数（IMX678 + 120° FOV）：
  δp = 0.5 pixel（匹配精度）
  f = 1200 pixel
  B = 80mm（推荐基线）

深度精度曲线：
  Z = 0.5m  → δZ = 12.5mm
  Z = 1.0m  → δZ = 50mm
  Z = 2.0m  → δZ = 200mm
  Z = 3.0m  → δZ = 450mm
  Z = 5.0m  → δZ = 1250mm（接近极限）

结论：有效工作距离集中在 0.3m–3.0m（完美匹配人形臂展+AMR避障）
```

## 附录B：软件API设计

### B.1 核心类：`StereoCamera`

```cpp
class StereoCamera {
public:
    struct Config {
        std::string exposure_mode = "auto";       // auto / manual / fixed
        uint32_t exposure_time_us = 33333;         // 30fps
        float analog_gain_db = 0.0f;
        std::string hdr_mode = "hdr_x2";           // off / hdr_x2 / hdr_x4
        std::optional<float> baseline_mm;
        float depth_min_m = 0.1f;
        float depth_max_m = 10.0f;
        float confidence_threshold = 0.65f;       // 默认阈值
    };

    // 创建设备实例
    static std::unique_ptr<StereoCamera> create(const Config&, rclcpp::Node*);

    // 获取单帧数据（含所有输出）
    FrameData grabFrame(uint32_t timeout_ms = 100);

    // 触发在线标定重收敛（Thor平台<1秒）
    void triggerRecalibration();

    // 获取设备状态
    DeviceStatus getStatus() const;

    // 获取标定参数（诊断用）
    CalibrationData getCalibration() const;

    // 获取最近一次错误
    ErrorInfo getLastError() const;
};
```

### B.2 核心结构：`FrameData`（自包含帧数据）

```cpp
struct FrameData {
    cv::Mat left_rect;     // 左目校正后图像

    struct Metadata {
        uint64_t hw_timestamp_ns;

        bool is_high_motion;
        float motion_score;
        uint8_t hdr_exposure_ratio;
        uint8_t left_valid_ratio;
        uint8_t right_valid_ratio;
        uint8_t overall_confidence;  // 0-100
        bool is_calibrated;
        float calib_quality;
    };
    Metadata metadata;
};
```
